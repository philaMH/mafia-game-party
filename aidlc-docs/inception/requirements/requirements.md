## 1. 프로젝트 개요

### 1.1 프로젝트 목적
사내 팀빌딩/워크숍에서 팀원들이 함께 즐길 수 있는 **마피아 게임 진행 보조 웹 애플리케이션**을 개발한다. 사용자들은 한 공간에 모여 대면으로 게임을 플레이하며, 시스템은 게임 마스터(GM) 역할을 자동으로 수행하여 사람 진행자 없이도 공정하고 재미있는 게임이 가능하도록 한다.

### 1.2 핵심 컨셉
- **오프라인 모임 보조 도구**: 모든 참가자가 한 공간에 모여 대면으로 플레이
- **시스템이 자동 GM**: 밤/낮 전환, 역할 배정, 투표 집계 등을 시스템이 자동 처리
- **진행자 음성 안내**: 공용 화면이 한국어 근엄한 톤으로 풍부한 진행자 멘트를 음성 출력 (브라우저 TTS)
- **모두가 플레이어**: 사람 진행자가 따로 없으며, 모든 참가자가 게임에 참여
- **하이브리드 화면 구조**: 공용 화면(공통 정보) + 개인 기기(비공개 정보)
- **로컬 LAN 실행**: 한 사람의 PC가 Go 서버 호스팅, 같은 Wi-Fi/LAN의 참가자들이 자기 기기로 접속
- **토론/대화는 현실에서**: 채팅·음성 등 인앱 통신 기능 없음

### 1.3 사용 맥락
- 회사 팀빌딩 / 워크숍 (부정기적, 한 공간에 모이는 모임)
- 6–12명 규모의 그룹

### 1.4 핵심 가치 (트레이드오프 우선순위)
**안정성과 신뢰성** — 게임 진행 중 끊김이나 데이터 유실로 인해 게임이 중단되지 않는 것을 최우선으로 한다.

## 2. 시스템 아키텍처 가정

### 2.1 실행 환경
- **호스트**: 한 사람의 PC에서 Go 서버 실행 (단일 바이너리)
- **클라이언트**: 같은 Wi-Fi/LAN의 다른 참가자들이 브라우저로 접속 (예: `http://192.168.x.x:8080`)
- **공용 화면**: 호스트 PC를 TV/프로젝터/큰 모니터에 연결하거나 별도 단말로 공용 화면 전용 페이지 접속

### 2.2 화면 구분
- **공용 화면 (Public View)**: 모두가 함께 보는 진행 정보 (밤/낮 단계, 투표 결과, 처형 결과, 게임 종료 등). **진행자 음성 안내(TTS)도 이 화면에서만 출력**.
- **개인 화면 (Player View)**: 각자 자기 기기에서만 보는 비공개 정보 (자기 역할, 키워드, 밤 행동 입력, 투표 입력). 음성 안내는 출력하지 않음.

### 2.3 기술 스택 (잠정)
| 구분 | 결정 | 비고 |
|---|---|---|
| 백엔드 언어 | **Go** | 사용자 선호 (Q10=E) |
| 클라이언트 | 웹 브라우저 기반 반응형 웹앱 | Q1 답변 |
| 실시간 통신 | WebSocket (잠정) | 다중 기기 상태 동기화 필요 |
| 데이터 저장 | 로컬 파일 또는 임베디드 DB (예: SQLite, BoltDB) | 로컬 실행이므로 외부 DB 불필요 |
| 배포 | 단일 바이너리 + 정적 자산 | LAN 호스트 PC에서 수동 실행 |

> 최종 기술 스택은 NFR 단계에서 확정한다.

## 3. 기능 요구사항 (Functional Requirements)

### FR-1. 게임방 관리

#### FR-1.1 게임방 생성
- 호스트가 서버 실행 후 첫 접속 시 게임방을 생성한다.
- 게임방은 1개의 활성 세션만 동시에 운영한다 (단일 게임 동시성).
- 게임방 생성 시 자동 생성된 접속 URL을 호스트가 다른 참가자들에게 안내한다.

#### FR-1.2 플레이어 입장
- 참가자는 닉네임만 입력하면 별도 인증 없이 입장한다.
- 닉네임 중복 시 서버가 에러를 반환하여 다른 닉네임 입력을 유도한다.
- 입장 후 대기실 화면에서 다른 참가자 목록을 확인할 수 있다.

#### FR-1.3 인원 제약
- **최소 인원**: 6명 (역할 분배가 의미 있는 최소치)
- **최대 인원**: 12명
- 호스트는 6명 미만에서는 게임을 시작할 수 없다.

#### FR-1.4 게임 시작
- 호스트(또는 첫 접속자)만 게임 시작 권한을 가진다.
- 시작 시 시스템이 역할과 키워드를 자동 배정한다.

### FR-2. 역할 시스템

#### FR-2.1 지원 역할
4종 역할을 지원한다:
| 역할 | 진영 | 능력 |
|---|---|---|
| **마피아 (Mafia)** | 마피아팀 | 밤마다 1명 지목하여 살해 |
| **시민 (Citizen)** | 시민팀 | 특수 능력 없음, 토론·투표 |
| **의사 (Doctor)** | 시민팀 | 밤마다 1명 지목하여 보호 (마피아 살해 무효화) |
| **경찰 (Police)** | 시민팀 | 밤마다 1명 지목하여 진영(마피아/시민) 확인 |

#### FR-2.2 역할 배분 규칙
인원수에 따라 다음 테이블로 역할을 자동 배분한다 (잠정안 — 게임 디자인 단계에서 조정 가능):

| 인원 | 마피아 | 의사 | 경찰 | 시민 |
|---|---|---|---|---|
| 6 | 1 | 1 | 1 | 3 |
| 7 | 2 | 1 | 1 | 3 |
| 8 | 2 | 1 | 1 | 4 |
| 9 | 2 | 1 | 1 | 5 |
| 10 | 3 | 1 | 1 | 5 |
| 11 | 3 | 1 | 1 | 6 |
| 12 | 3 | 1 | 1 | 7 |

#### FR-2.3 역할 비공개
- 역할은 각자 자기 기기 화면에서만 확인 가능하다.
- 공용 화면에는 누가 어떤 역할인지 절대 노출되지 않는다 (게임 종료 시까지).

### FR-3. 자기소개 키워드 시스템

#### FR-3.1 키워드 생성
- 시스템은 매 게임마다 무작위로 키워드를 부여한다 (Clarification CQ3-1=B).
- 키워드는 역할별로 사전 정의된 키워드 풀에서 랜덤 추출한다.
- 같은 역할 내 여러 명이 있는 경우, 같은 키워드를 부여할지 / 서로 다른 키워드를 부여할지는 디자인 단계에서 결정 (기본 가정: 동일 역할은 동일 키워드).

#### FR-3.2 키워드 표시
- 키워드는 자기 기기 화면에서만 비공개로 확인 가능하다.
- 첫째날 낮 자기소개 단계 시작 시점에 화면에 표시된다.

#### FR-3.3 키워드 검증/페널티
- **검증 방식**: 시스템 검증 없음. 단순 가이드만 제공 (Clarification CQ3-2=A).
- **페널티**: 페널티 없음 — 단순히 의심 단서로만 활용된다 (Clarification CQ3-3=A).
- 즉, 시스템은 키워드를 화면에 노출만 하고, 발화 여부는 다른 플레이어들의 기억과 직관에 의존한다.

### FR-4. 낮/밤 사이클

#### FR-4.1 첫째날 (Day 1)
- **자기소개 단계**: 모든 살아있는 플레이어가 한 명씩 자기소개 (현실 발화). 자기 화면의 키워드를 발화하도록 가이드.
- 첫째날엔 투표/처형 없음 (잠정 — 게임 디자인에서 조정 가능).

#### FR-4.2 밤 단계 (Night)
- 공용 화면에 "밤" 표시.
- 살아있는 마피아·의사·경찰이 자기 기기에서 비공개로 행동 입력:
  - 마피아: 살해 대상 1명 지목 (다수의 마피아 시 다수결 또는 단일 합의 — 디자인에서 결정)
  - 의사: 보호 대상 1명 지목 (자기 자신 가능 여부도 디자인에서 결정)
  - 경찰: 조사 대상 1명 지목 → 자기 화면으로 결과(마피아/시민) 확인
- 모든 행동 완료 시 자동으로 낮으로 전환.

#### FR-4.3 낮 단계 (Day)
- 공용 화면에 "낮" 표시 + 밤 결과 발표 (사망자 또는 "평화로운 밤" 표시).
- **토론 시간**: 현실에서 자유 토론 (시스템은 타이머만 제공).
- **투표 단계**: 자기 기기에서 살아있는 플레이어 1명에게 비공개 투표.
- 시스템이 집계 후 공용 화면에 결과 발표 → 최다 득표자 처형.
- 동률 시 처리 규칙 (잠정안):
  - **재투표 1회** → 그래도 동률이면 **처형 없이 다음 밤으로 진행**.
  - 디자인 단계에서 확정.

#### FR-4.4 사이클 반복
- 사망자 발생 시 즉시 게임 종료 조건 검사 → 미충족이면 다음 밤으로 진행.
- 종료 조건 충족 시 게임 종료.

### FR-5. 게임 종료 조건

| 조건 | 결과 |
|---|---|
| 마피아 인원 = 0 | **시민팀 승리** |
| 마피아 인원 ≥ 시민 진영 인원 | **마피아팀 승리** |

- 시스템이 매 단계마다 자동 판정.
- 종료 시 공용 화면에 승리팀 + 모든 플레이어의 역할 공개.

### FR-6. 게임 결과 저장

#### FR-6.1 저장 범위
다음 항목을 게임 종료 시 저장한다 (Q8=B 결과 요약 수준):
- 게임 ID, 시작/종료 시각
- 참가자 닉네임 + 배정 역할
- 승리팀
- 각 플레이어의 최종 생존 여부

#### FR-6.2 저장 위치
- 호스트 PC의 로컬 파일 또는 임베디드 DB.
- 외부 네트워크 전송 없음.

#### FR-6.3 조회
- 호스트가 과거 게임 결과 목록을 조회할 수 있다.
- 단순 통계(누적 게임 수, 닉네임별 승률 등)는 향후 확장 시 추가 (FR-7과 연결).

### FR-7. 향후 확장 고려 (점진적)

> Q13=B 점진적 확장 의도에 맞춰 다음 항목을 **확장 가능한 구조**로 설계한다 (현재 단계에서 구현하지는 않음):
- 역할 추가 (도둑, 군인, 변호사 등)
- 키워드 풀 외부 파일 분리 (사용자 편집 가능)
- 게임 옵션 (첫째날 투표 활성화, 의사 자가 보호 허용 등)
- 누적 통계 화면

### FR-8. 진행자 음성 안내 (TTS)

#### FR-8.1 TTS 엔진
- 클라이언트 측 **브라우저 내장 Web Speech API (`SpeechSynthesis`)** 를 사용한다 (Clarification CR2-Q1=A).
- 외부 인터넷 연결, 별도 음원 파일, 서버 측 TTS 라이브러리 없이 동작.
- 서버는 어떤 메시지를 발화할지 텍스트로 전송하고, 클라이언트가 로컬 TTS로 발화한다.

#### FR-8.2 출력 위치
- 음성은 **공용 화면(`/public` 등 공용 페이지) 클라이언트에서만** 출력한다 (Clarification CR2-Q2=A).
- 개인 화면(자기 기기)에서는 음성 출력을 비활성화한다 (에코·시간차·산만함 방지).

#### FR-8.3 음성 특성
| 속성 | 값 |
|---|---|
| **언어** | 한국어 (`ko-KR`) 만 지원 |
| **톤** | **근엄한 톤** — 사용자 자유 의견 반영. 낮은 피치(pitch ≈ 0.8) + 약간 느린 속도(rate ≈ 0.9)로 잠정 매핑 |
| **음성 선택** | 브라우저가 제공하는 한국어 음성 중 가능한 가장 차분/저음 보이스를 자동 선택. 가용 음성이 없으면 시스템 기본 한국어 음성 사용 |

#### FR-8.4 안내 범위 (풍부한 진행 안내)
**Clarification CR2-Q3=C** — 단계 전환 + 결과 발표 + 진행자 멘트까지 폭넓게 안내한다. 최소한 다음 시점에 발화한다 (실제 문구는 디자인 단계에서 확정):

| 시점 | 안내 예시 (잠정) |
|---|---|
| 게임 시작 | "마피아 게임이 시작됩니다. 모두 자신의 화면을 확인하세요." |
| 첫째날 자기소개 시작 | "첫째날 낮입니다. 차례대로 자기소개를 해주세요." |
| 밤 진입 | "밤이 깊어졌습니다. 모두 눈을 감으세요." |
| 마피아 차례 | "마피아는 살해할 대상을 지목하세요." |
| 의사 차례 | "의사는 보호할 사람을 선택하세요." |
| 경찰 차례 | "경찰은 조사할 사람을 선택하세요." |
| 낮 진입 + 사망 발표 | "아침이 밝았습니다. 지난밤 ○○○님이 사망했습니다." |
| 평화로운 밤 | "지난밤은 평화로웠습니다." |
| 토론 시작 | "토론을 시작합니다." |
| 토론 카운트다운(잔여 30초) | "토론 종료까지 30초 남았습니다." |
| 투표 시작 | "투표를 시작합니다. 처형할 사람을 선택하세요." |
| 투표 결과 발표 | "투표 결과 ○○○님이 처형되었습니다." 또는 "투표가 동률입니다. 재투표를 진행합니다." |
| 게임 종료 | "게임이 종료되었습니다. ○○팀의 승리입니다." |

#### FR-8.5 음성 ON/OFF 토글
- 공용 화면에 음성 안내 **ON/OFF 토글**을 제공한다 (기본값: ON).
- 토글 상태는 게임 세션 동안 유지되며, 새 게임 시작 시 기본값(ON)으로 복원된다.

#### FR-8.6 큐잉 / 인터럽션 정책
- 짧은 시간에 여러 안내가 연속으로 발생할 경우, 클라이언트에서 **순차 재생(큐잉)** 한다 (겹쳐서 출력 금지).
- 단계가 전환되어 이전 안내가 무의미해진 경우(예: 토론 카운트다운 중 게임이 강제 종료), **현재 발화를 중단하고 새 안내로 대체**한다.

#### FR-8.7 자막 동시 표시
- 모든 음성 안내는 **공용 화면에 동일한 텍스트가 함께 표시**된다 (소리 환경이 안 좋거나 발화가 불분명한 경우 대비).

## 4. 비기능 요구사항 (Non-Functional Requirements)

### NFR-1. 안정성 / 신뢰성 (최우선)
| 항목 | 요구사항 |
|---|---|
| **상태 복원** | 서버 재시작 시 진행 중인 게임 상태를 복구할 수 있어야 한다 (디스크 영속화) |
| **재연결** | 클라이언트가 LAN 끊김 후 재접속 시 본인 역할/키워드/현재 단계로 자동 복원 |
| **데이터 무결성** | 투표·밤 행동 입력은 서버 측에서 일관성 검증 후 적용 (이중 입력, 사망자 입력 등 차단) |
| **단일 GM 보장** | 동시에 여러 명이 호스트 권한을 행사할 수 없도록 락 처리 |

### NFR-2. 성능
| 항목 | 요구사항 |
|---|---|
| **동시 접속** | 12명 + 공용 화면 1개 = 약 13개 클라이언트 동시 처리 무리 없음 |
| **응답 시간** | 사용자 행동(투표, 행동 입력) → 서버 반영 < 500ms (LAN 환경 가정) |
| **상태 동기화** | 단계 전환(밤→낮 등)은 모든 클라이언트에 1초 이내 반영 |

### NFR-3. 사용성 (Usability)
| 항목 | 요구사항 |
|---|---|
| **반응형 UI** | 스마트폰·노트북·태블릿 브라우저 모두 동작 |
| **공용 화면 가독성** | 멀리서도 보이는 큰 폰트와 명확한 단계 표시 |
| **명확한 가이드** | 각 단계마다 다음에 뭘 해야 하는지 화면에 안내 |

### NFR-4. 보안
| 항목 | 결정 |
|---|---|
| **Security Baseline 확장** | **비활성화** (Q14=B). 사내망 + 일회성 게임 환경의 PoC/사내 도구 성격 반영 |
| **최소한의 보호** | 자기 역할/키워드는 자기 기기 외에 노출되지 않음 (게임 무결성 차원) |
| **외부 노출 없음** | 외부 인터넷 공개 없음 (LAN 한정) |

### NFR-5. 호환성
- 최신 Chrome / Edge / Safari / Firefox 지원.
- 모바일 브라우저(iOS Safari, Android Chrome) 지원.
- **공용 화면**은 **Web Speech API (`SpeechSynthesis`)** 를 지원하는 브라우저여야 한다 (Chrome/Edge/Safari 권장. Firefox는 한국어 음성 가용성이 OS 의존적이라 호스트 PC OS에 한국어 음성 팩이 설치되어 있어야 함).
- Web Speech API 미지원/한국어 음성 부재 시 자막 표시는 정상 동작해야 하며, 시스템은 토스트/배지로 "음성 사용 불가" 상태를 알린다.

### NFR-6. 유지보수성 / 확장성
- 역할·키워드를 코드 외부 자원으로 분리 (점진적 확장 대비).
- 게임 진행 단계(state machine)를 명시적으로 설계해 단계 추가/변경이 용이하도록.

### NFR-7. 운영 단순성
- 호스트가 단일 바이너리 실행만으로 게임 시작 가능 (별도 DB 설치, 환경변수 설정 등 불필요).
- 게임 종료 후 종료 명령 한 번으로 깔끔히 종료.

## 5. 사용자 시나리오 (User Scenarios)

### 시나리오 1: 정상 게임 진행
1. 호스트가 자기 노트북에서 Go 바이너리 실행 → 콘솔에 `http://192.168.0.10:8080` 표시.
2. 호스트는 노트북을 회의실 모니터+스피커에 연결하여 공용 화면을 띄움 (`/public` 페이지). 공용 화면은 한국어 TTS 가용 여부를 자동 점검한다.
3. 8명의 팀원이 각자 스마트폰으로 위 URL에 접속, 닉네임 입력 후 입장.
4. 호스트가 "게임 시작" 클릭 → 시스템이 마피아 2 / 의사 1 / 경찰 1 / 시민 4 자동 배정 + 키워드 부여.
   - **공용 화면 음성**: "마피아 게임이 시작됩니다. 모두 자신의 화면을 확인하세요."
5. 각 팀원은 자기 폰 화면에서 자기 역할과 키워드 확인.
6. 공용 화면에 "Day 1 — 자기소개" 표시 + **음성**: "첫째날 낮입니다. 차례대로 자기소개를 해주세요." → 한 명씩 자기소개 (현실 발화, 자기 키워드 발화 시도).
7. 자기소개 완료 후 호스트가 "밤" 진행 → 공용 화면 "Night 1" + **음성**: "밤이 깊어졌습니다. 모두 눈을 감으세요." → 이어서 "마피아는 살해할 대상을 지목하세요." 등 차례별 안내.
8. 마피아·의사·경찰은 자기 폰에서 행동 입력 (자기 폰에서는 음성이 나오지 않음).
9. 공용 화면에 사망자 표시 + **음성**: "아침이 밝았습니다. 지난밤 김철수님이 사망했습니다."
10. 토론(타이머 + 잔여 30초 음성 안내) → 투표 → 처형 발표(음성) → 다음 밤 …
11. 시민팀 승리 → 공용 화면에 모든 역할 공개 + **음성**: "게임이 종료되었습니다. 시민팀의 승리입니다." + 결과 저장.

### 시나리오 2: 클라이언트 재접속
- 한 명의 폰이 Wi-Fi에서 떨어졌다가 재접속할 경우, 닉네임으로 자동 식별되어 본인 화면(역할/키워드/현재 단계)이 복원된다.

### 시나리오 3: 호스트 PC 재시작
- 호스트 PC가 도중에 재부팅된 경우, 서버를 다시 띄우면 마지막 진행 상태로 복원된다.
- 클라이언트들은 재접속하여 게임 이어가기 가능.

### 시나리오 4: 인원 미달
- 6명 미만이 입장한 상태에서 게임 시작을 누르면 에러 메시지 표시 (`최소 6명 필요`).

### 시나리오 5: 동률 투표
- 낮 투표 결과 동률 → **공용 화면 음성**: "투표가 동률입니다. 재투표를 진행합니다." → 자동 재투표 1회 → 그래도 동률이면 처형 없이 다음 밤으로.

### 시나리오 6: 한국어 TTS 음성 부재
- 공용 화면이 시작 시 Web Speech API에서 한국어 음성을 발견하지 못한 경우, 화면에 "음성 사용 불가 — 자막만 표시됩니다" 토스트를 띄우고, 안내는 자막으로만 진행한다. 게임 진행 자체는 영향받지 않는다.

### 시나리오 7: 음성 OFF
- 호스트가 공용 화면의 음성 토글을 OFF로 전환 → 이후 모든 안내는 자막으로만 표시. 토론 카운트다운 등 시간성 안내는 시각적 강조(깜빡임 등)로 대체된다.

## 6. 가정 및 제약 사항 (Assumptions & Constraints)

### 6.1 가정
- 모든 참가자는 같은 Wi-Fi/LAN에 접속할 수 있다.
- 호스트 PC가 게임 진행 동안 켜져 있다.
- 모든 참가자는 자기 기기(스마트폰 또는 노트북)를 가지고 있다.
- 토론/자기소개 등 발화는 현실 공간에서 이루어진다.
- 공용 화면을 띄우는 호스트 PC에 **스피커가 연결되어 있고 회의실에서 들릴 만큼 충분한 볼륨**이 가능하다.
- 호스트 PC OS/브라우저에 **한국어 TTS 음성이 사용 가능**하다 (대부분 최신 Windows/macOS는 기본 제공).

### 6.2 제약
- 외부 인터넷 공개 배포는 범위 외.
- 인앱 채팅/음성 통신은 범위 외 (단, **시스템→사용자 단방향의 진행자 음성 안내**는 제공).
- 사용자 인증·계정 시스템은 범위 외 (닉네임만).
- Security Baseline 확장 미적용 → 운영 환경 수준의 보안 검증 범위 외.
- 음성 안내는 **한국어만** 지원 (다국어 지원은 향후 확장 범위).

## 7. 핵심 요구사항 요약

- **누가**: 6–12명의 회사 팀원
- **언제**: 사내 팀빌딩/워크숍 등 한 공간에 모인 자리
- **무엇을**: 마피아·시민·의사·경찰 4역할 마피아 게임
- **어떻게**:
  - 한 사람 PC가 Go 서버 + 공용 화면
  - 다른 사람들은 자기 기기로 LAN 접속 (비공개 정보 확인용)
  - 시스템이 자동 GM (밤/낮 전환, 투표 집계)
  - **공용 화면이 한국어 근엄한 톤으로 진행자 음성 안내 출력 (브라우저 TTS, 자막 동시 표시)**
  - 토론·자기소개는 현실에서
  - 자기소개 단계에 역할별 무작위 키워드 발화 가이드 (검증·페널티 없음)
- **왜**: 사람 진행자 부담 없이 공정하게, 끊김 없이 즐기기 위해
- **트레이드오프 우선순위**: 안정성/신뢰성 > 그 외
