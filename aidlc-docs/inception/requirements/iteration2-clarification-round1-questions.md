# Iteration 2 — Requirements Clarification Round 1

**Purpose**: Iteration 2 Intake 답변에서 발견된 모호성/모순을 해소하기 위한 1차 명확화 질문.

**현재 시스템 As-Is 요약 (요구사항 v1.1 + 코드 기준)**:
- 게임방은 서버 시작 시 자동 1개만 운영 (단일 active session)
- "호스트(또는 첫 접속자)" 가 게임 시작 권한 보유 — **호스트도 플레이어로 참여**
- 첫째날 자기소개는 GM 화면에서 "다음 발언자" 버튼을 호스트가 눌러 턴을 진행
- 인원 6–12명, 인원수에 따라 자동 역할 배분 테이블 사용 (예: 6명 → 마피아 1 / 의사 1 / 경찰 1 / 시민 3)
- TTS 음성 안내는 공용 화면(`/public`)에서 재생

**작성 규칙**:
- 각 질문 아래 `[Answer]:` 줄에 알파벳(A~E) 또는 자유 서술 기재
- "Other" 선택 시 `[Answer]: E — 설명...` 형식으로 자유 서술
- 모두 작성 후 채팅창에 "완료" 라고 적어 주시면 다음 단계로 진행합니다.

## CR1-Q1. 호스트의 "방 개설" 동작 시점

호스트가 /public 에서 방을 개설한다고 하셨는데, 현재는 서버가 시작되면 자동으로 단일 세션이 생기고 누구든 첫 접속자가 호스트가 되는 구조입니다. 새로운 흐름을 어떻게 정의할까요?

A) **호스트가 /public 접속 시 "방 개설" 버튼을 명시적으로 누르고, 그 전에는 플레이어가 /play 로 들어와도 "방이 아직 없습니다" 표시** — 가장 명시적
B) **서버 시작 시 빈 세션 자동 생성, 호스트가 /public 접속하여 "게임 설정" 입력 후 "참가 모집 시작" 누르면 플레이어 입장 가능** — 설정 게이트 형태
C) **호스트의 첫 /public 접속이 곧 방 개설로 간주, 게임 설정은 별도 모달/패널에서 입력** — 가장 가벼운 변경
E) Other (자유 서술)

[Answer]: A 

---

## CR1-Q2. "서버는 1개 방만 개설" 의 강제 수준

A) **하드 락** — 이미 방이 있으면 두 번째 호스트가 /public 에 접속해도 새 방을 개설할 수 없고, 기존 방에 GM 권한 없이 read-only 관전자로 진입
B) **하드 락 + 거부** — 이미 방이 있으면 두 번째 호스트의 /public 접속은 "이미 다른 호스트가 방을 운영 중" 안내만 보여주고 진입 차단
C) **소프트 — 마지막 호스트 우선** — 방을 새로 개설하면 기존 방은 종료되고 모든 플레이어가 LOBBY로 돌아감 (덮어쓰기 동작)
E) Other

[Answer]: B

---

## CR1-Q3. 호스트는 플레이어가 절대 될 수 없는가?

A) **완전 분리** — 호스트로 입장하면 이후 어떤 경우에도 그 브라우저는 플레이어로 게임에 참여할 수 없음(참가자 수에서 제외, 역할 미배정)
B) **선택 가능** — 호스트가 본인을 플레이어로 포함시킬지 옵션을 제공 (체크박스 등). 기본값은 "GM 전용"
C) **단일 화면 전환 가능** — 호스트가 방을 개설한 뒤 별도로 본인의 다른 기기(휴대폰 등)에서 /play 로 접속하여 플레이어로도 참여 가능 (호스트 화면과 별도)
E) Other

[Answer]: C

---

## CR1-Q4. 게임 설정 — "최대 참여 인원"

A) **현재 그대로 6–12명 범위만 허용** — 호스트가 6, 7, 8 ... 12 중 한 값 선택. 6 미만/12 초과는 비허용
B) **하한 6명 유지, 상한 호스트 임의 (예: 16, 20)** — 추가 인원에 대한 역할 배분은 새 규칙 필요
C) **하한도 호스트 임의 (예: 4명도 가능)** — 소규모 테스트용으로 허용
E) Other

[Answer]: A

---

## CR1-Q5. 게임 설정 — "마피아 수"

A) **호스트 임의 지정 (마피아 수만)** — 호스트가 마피아 수를 임의 선택, 의사/경찰은 현재 규칙(각 1명 고정) 유지, 나머지가 시민
B) **호스트가 모든 역할 수를 임의 지정** — 마피아/의사/경찰/시민 수를 모두 호스트가 슬라이더로 지정 (총합 = 참여 인원)
C) **추천값 제공 + 마피아 수만 미세 조정** — 시스템이 인원수 기반 추천(현재 테이블)을 띄우고 호스트가 마피아 수만 ±1 조정
E) Other

[Answer]: C

---

## CR1-Q6. 마피아 수 게임 밸런스 검증 (가드레일)

호스트가 임의 마피아 수를 지정할 때 게임이 깨질 수 있습니다(예: 6명 중 마피아 5명). 어디까지 허용할까요?

A) **무제한** — 호스트의 의도를 신뢰, 검증 없이 시작 허용
B) **마피아 수 < 시민팀 수 강제** — 마피아가 시민팀(시민+의사+경찰)보다 적어야 시작 가능
C) **권장 범위 제시 + 경고만** — 권장치(현재 자동 배분 규칙) 벗어나면 "비추천" 경고 표시 후 호스트가 강행 가능
E) Other

[Answer]: C

---

## CR1-Q7. 자기소개 단계의 "종료" 액션을 누가 누르는가?

A) **자기소개 중인 플레이어 본인** — 자기 기기(/play)에 "내 자기소개 종료" 버튼이 활성화되어 본인이 직접 누름
B) **호스트가 누름 (현재와 동일)** — 호스트가 /public 에서 "다음 발언자" 클릭, 단지 누구를 다음으로 할지 호스트가 선택
C) **자동 타이머** — 인당 N초 경과하면 자동 다음 순번 (호스트는 "조기 종료" 만 누름)
E) Other

[Answer]: A

---

## CR1-Q8. 다음 자기소개 발언자 선정 방식

Q6 자유서술 답변에 "호스트가 다음 순번을 지정" 과 "호스트는 직접 조작하지 않도록" 이 다소 상충됩니다. 어느 쪽으로 정리할까요?

A) **자동 라운드 로빈** — 시스템이 입장 순(또는 알파벳 순)으로 자동 다음 발언자 결정, 호스트는 화면을 모니터링만 함 (강제 종료 외 조작 없음)
B) **현재 발언자가 다음 발언자를 지명** — 발언 종료 시 자기 화면에서 다음 사람을 직접 선택
C) **호스트가 다음 사람 선택** — 호스트 화면에 "다음 발언자: [드롭다운]" UI, 호스트가 매 턴 선택. (이 경우 호스트가 GM 으로서 매 턴 1번 클릭은 필요)
E) Other

[Answer]:  A

## CR1-Q9. 호스트가 게임 진행 중 누르는 버튼의 최종 범위

"호스트는 강제 종료 같은 동작이 아니면 사람이 직접 조작하지 않도록 하고 싶음" 답변을 어디까지 적용할까요? 다음 동작들 중 호스트가 **직접 클릭해야 하는 것** 만 골라 주세요 (복수 선택 가능, 표기: A,B,C 식).

후보:
- ㉠ **방 개설 / 게임 설정 입력**
- ㉡ **게임 시작** (최소 인원 충족 후)
- ㉢ **자기소개 단계 시작**
- ㉣ **자기소개 다음 발언자 선택** (CR1-Q8 답변에 따라 자동/수동 결정)
- ㉤ **밤 단계 시작 / 종료**
- ㉥ **낮 단계 시작 / 종료 (토론·투표 시작)**
- ㉦ **투표 결과 발표 / 처형 진행**
- ㉧ **게임 강제 종료 / 무효화**
- ㉨ **게임 결과 공개 (밤·낮 결과 반영 화면 전환)**

A) **㉠ + ㉧ 만** — 방 개설과 강제 종료 외에는 모두 자동
B) **㉠ + ㉡ + ㉧ 만** — 게임 시작은 호스트가 명시적으로 누름, 그 외 모두 자동
C) **㉠ + ㉡ + ㉣ + ㉧ 만** — 게임 시작과 자기소개 발언자 선택까지 호스트가 누름
E) **자유 조합** — 후보 ㉠~㉨ 중 클릭 필요한 것만 직접 표기 (예: `E — ㉠,㉡,㉥,㉧`)

[Answer]: B)

## CR1-Q10. 호스트 권한 인증 / 보호

호스트만 방 개설·강제 종료 권한을 갖는다면 다른 사람이 호스트 페이지(/public)로 접속하지 못하게 보호할 필요가 있을 수 있습니다.

A) **무인증 (현재 그대로)** — /public URL을 아는 사람이 누구든 호스트 화면 접근 가능 (사내 LAN 신뢰)
B) **간단한 호스트 비밀번호** — 서버 시작 시 콘솔에 출력된 6자리 코드로 /public 진입 가능
C) **첫 번째 /public 접속자만 호스트** — 이후 /public 접속자는 read-only 관전자
E) Other

[Answer]: C

## CR1-Q11. 본 반복에서 변경하는 화면/문구의 한국어 톤

호스트 GM 화면 신규 UI(방 개설, 게임 설정 등)의 한국어 문구 톤은 어떻게 할까요?

A) **현재 톤 유지** (예: "참가자 모집 중", "게임 시작") — 일관성 우선
B) **사회자/진행자 톤으로 통일** ("방을 개설합니다", "참가자를 받습니다", "설정을 마쳤다면 게임을 시작합니다") — 호스트가 사회자 역할임을 강조
E) Other

[Answer]: B
