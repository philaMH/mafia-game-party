## 1. 매핑 원칙

- **Primary (P)**: 해당 요구사항의 **주요 책임 단위**. 구현·테스트의 본진.
- **Secondary (S)**: **보조 단위**. UI 표시, 데이터 통과, 트리거 등 부분 책임.
- 한 요구사항은 **최소 1 Primary** 단위를 가져야 함.

## 5. 시나리오 ↔ 단위 매핑 (requirements.md §5)

| 시나리오 | 핵심 흐름 | 관여 단위 |
|---|---|---|
| 시나리오 1: 게임 시작 → 자기소개 → 야간 진입 | 호스트 시작 → 역할 배분 → 키워드 부여 → TTS 안내 | U1, U2, U3, U5 |
| 시나리오 2: 클라이언트 단절 후 재연결 | WS 끊김 → 재접속 → 본인 상태 복원 | U2, U3, U5 |
| 시나리오 3: 호스트 PC 재시작 후 진행 상태 복원 | 바이너리 재기동 → LoadActiveSnapshot → Restore | U2, U1, U4 |
| 시나리오 4: 동률 처리 | 투표 동률 → 재투표 1회 → 무처형 → NIGHT | U1 (+ U5 표시) |
| 시나리오 5: 게임 종료 → 결과 저장·조회 | GameEnded → SaveResult → ListResults | U2 (+ U5 결과 화면) |
| 시나리오 6: TTS 부재 환경 | Web Speech API 부재 감지 → 자막 폴백 | U5 |
| 시나리오 7: TTS OFF | 사용자 토글 → SpeechSynthesis 호출 차단 | U5 |
